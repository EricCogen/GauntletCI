## [2.1.1] - 2026-04-28

### Changed
- **NuGet README decoupled**: Added `nuget-readme.md` as a NuGet-specific README using plain markdown and absolute image URLs for correct rendering on NuGet.org. `README.md` unchanged for GitHub.
- **Resized NuGet logo**: Added `GauntletCI-nuget.png` (200x262) for use in the NuGet package README.
- **GCI0045 recall improvement**: Rule now fires when a new interface has 0 visible implementers in the diff (not just exactly 1). Change: `implCount != 1` guard replaced with `implCount > 1`. Interfaces added with no implementation in the same PR are often premature abstraction. Finding message distinguishes "no visible" from "exactly one" implementer. Added `NewInterfaceWithNoImplementorInDiff_ShouldFire` regression test. 1191 tests pass.
- **GCI0045 labeler false positives**: Heuristic changed from `addedLines` (all files, all extensions) to `prodCsLines` (non-test `.cs` files only) to mirror the rule's own filtering. Previously, interface definitions in `.txt` API-approval snapshots and test files created false positive expected labels that the rule correctly never fires on - pulling down apparent recall from labeler noise, not real rule misses.
- **GCI0032 rule bug - removed assertion lines**: The throw-assertion check now filters out `DiffLineKind.Removed` hunk lines from test files. Previously a deleted `Assert.Throws` line (evidence that coverage was removed) incorrectly suppressed the finding. Added `ThrowNewWithRemovedAssertionInTestFile_ShouldFlag` regression test.
- **GCI0032 labeler false positives**: Heuristic rewritten to mirror rule behavior exactly via per-file `rawDiff` iteration: (1) throw detection restricted to added lines in non-test `.cs` files only (was `addedLines` across all files, causing 41 FP labels from `throw new` in test files); (2) assertion check now scans both context and added lines in test `.cs` files (matching rule's hunk-line scan), suppressing the label when existing tests already cover the exception type; previously no assertion check existed, causing 33 FP labels where prod throws had corresponding test assertions.
- **GCI0038 false positives (round 2)**: Three precision fixes: (1) `CheckDirectInstantiation` now skips infrastructure files (`Program.cs`, `Startup.cs`, `*Extensions.cs`, paths containing `ServiceCollection`) - the same guard already applied in `CheckServiceLocator`; DI registration code legitimately instantiates concrete types. (2) `CheckCaptiveDependency` now skips infrastructure files and test files - mixed-lifetime registrations in DI setup files and test overrides are intentional, not bugs. (3) `DirectInstantiationExclusions` gains `"EventHandler("` and `"+= new "` to stop matching WPF/WinRT event handler delegates (`RoutedEventHandler`, `RequestNavigateEventHandler`, etc.) that hit the `Handler` suffix pattern. Added 4 regression tests: `DirectInstantiation_InInfrastructureFile_ShouldNotFlag`, `CaptiveDependency_InInfrastructureFile_ShouldNotFlag`, `CaptiveDependency_InTestFile_ShouldNotFlag`, `DirectInstantiation_EventHandlerDelegate_ShouldNotFlag`. 1197 tests pass.
- **GCI0038 rule bug - service locator in test files**: `CheckServiceLocator` now excludes test files (same guard as `CheckDirectInstantiation`). ABP-style test bases use `GetRequiredService<T>()` as test scaffolding - not a DI violation. Added `ServiceLocator_InTestFile_ShouldNotFlag` regression test.
- **GCI0038 labeler false positives**: GCI0038 block rewritten from a global `addedLines.Any(...)` heuristic to per-file `rawDiff` iteration. Service locator and direct instantiation patterns are now checked only on added lines in non-test, non-infra `.cs` files. The old labeler used `.GetRequiredService<` (dot prefix) which matched bare `GetRequiredService<>()` calls in ABP test base classes - the rule requires a `provider.`/`serviceProvider.`/`_serviceProvider.` prefix. Patterns now aligned with the rule's `ServiceLocatorPatterns` array. Also adds `Client` suffix to the injectable regex to match the rule. 1193 tests pass.
- **`action.yml` Marketplace readiness**:Input `commit` default changed from an illegal expression (`${{ github.event.pull_request.head.sha || github.sha }}`) to `""`. GitHub Actions does not evaluate expressions in `action.yml` defaults; the raw string was passed as the commit SHA. The run step now resolves the SHA via `[ -z "$COMMIT" ] && COMMIT="${{ ... }}"`. Default `gauntletci-version` bumped from `2.0.0` to `2.1.0`.
- **GCI0029 false positives (round 2)**: Removed `fullname` from `PiiTerms`. `Type.FullName`, `Assembly.FullName`, and `FileInfo.FullName` are ubiquitous .NET reflection properties - not person names. All 8 corpus triggers from `apache_logging-log4net_pr201` were `converterType.FullName`, `objectType?.FullName`, `configFile.FullName`, etc. Added `TypeFullNameInLogCall_ShouldNotFlag` regression test. 1190 tests pass.
- **GCI0010 labeler over-labeling**: Removed `config` and `environment variable` from the GCI0010 `CommentRules` keyword set. Both were too broad - any review comment mentioning config changes (unrelated to hardcoding) caused spurious positive labels. Retained `hardcoded`, `hard-coded`, `magic string`, and `magic number` as the signal set.

### Added
- **PR description quality enricher** (`corpus pr-description enrich`): `PRDescriptionEnricher` fetches PR title and body from the GitHub PR API and computes quality signals: `title_length`, `body_length`, `is_empty_body` (body null or < 30 chars), `has_linked_issue` (body matches "Fixes/Closes/Resolves #N" or full issue URL), `has_wip_keywords` (title or body contains "wip", "work in progress", "draft", "do not merge", "dnm", "temp", "hack", "fixme", "todo"), and `label_count`. Results written to `pr_description_enrichments`. Fixtures with WIP keywords + empty body + low social validation trigger `UNVALIDATED_BEHAVIORAL_RISK`. Options: `--fixtures`, `--db`, `--tier`, `--delay-ms`, `--limit`. Reference: Zanoni et al. 2018.
- **Author experience enricher** (`corpus author-experience enrich`): `AuthorExperienceEnricher` fetches the PR author login from the PR API, estimates total commit count to the repo via the commits API pagination Link header (capped at 1000), and checks the contributors list to detect first-time contributors. Computes `experience_tier`: "none" (0 commits), "low" (1-5), "medium" (6-50), "high" (51+). Results written to `author_experience_enrichments`. First-time contributors with empty PR body amplify `UNVALIDATED_BEHAVIORAL_RISK` classification. Contributor list is cached in-memory per repo. Options: `--fixtures`, `--db`, `--tier`, `--delay-ms`, `--limit`. References: Kamei et al. 2013 EXP/REXP features; Foucault et al. 2015 code ownership.
- **`pr_description_enrichments` DB table**: Migration adds `pr_description_enrichments (fixture_id, repo, title_length, body_length, is_empty_body, has_linked_issue, has_wip_keywords, label_count, fetched_at_utc)` with `PRIMARY KEY (fixture_id)`.
- **`author_experience_enrichments` DB table**: Migration adds `author_experience_enrichments (fixture_id, repo, author_login, commit_count, is_first_contributor, experience_tier, fetched_at_utc)` with `PRIMARY KEY (fixture_id)`.
- **Test coverage gap enricher** (`corpus test-coverage enrich`): `TestCoverageEnricher` parses changed file paths from `diff.patch` and classifies each `.cs` file as production (not generated, not in test folder) or test (filename ends in `Tests.cs`/`Test.cs`/`Specs.cs` or path contains `test`/`tests`/`spec`/`specs` folder). Computes `test_coverage_gap` (prod changes with no test changes) and `test_to_prod_ratio`. Results written to `test_coverage_enrichments`. Fixtures with a coverage gap and social score < 0.5 contribute to `UNVALIDATED_BEHAVIORAL_RISK` classification. No HTTP client or GitHub API needed. Options: `--fixtures`, `--db`, `--tier`, `--limit`.
- **Diff entropy enricher** (`corpus diff-entropy enrich`): `DiffEntropyEnricher` computes Kamei et al. (2013) JIT defect prediction features from `diff.patch`: NF (file count), ND (distinct parent directories), NS (distinct top-level namespaces estimated from first two path segments), total lines changed, Shannon change entropy (bits), and normalized entropy in [0,1].
