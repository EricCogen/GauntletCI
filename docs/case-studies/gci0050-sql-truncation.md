# Case Study: GCI0050 - SQL Column Truncation Data Loss

## The Problem

Column truncation during schema migrations silently loses data. When a database migration reduces the size of a string column (e.g., VARCHAR(255) -> VARCHAR(50)), any existing values longer than the new limit are truncated. The database doesn't error—it just cuts the data. No alerts, no rollback, no warnings. The data is gone, and the application keeps running.

This happens most often during:
- Refactoring field lengths to "save space"
- Removing redundant fields by merging them
- Normalizing data models without checking existing data

## Real-World Failure

**E-commerce platform data loss (2019)**: A team reduced the `phone_number` column from VARCHAR(30) to VARCHAR(10) for "consistency." Thousands of international phone numbers (many with country codes, extensions, etc.) were silently truncated. Customer support couldn't reach customers. Billing and shipping addresses were orphaned. Recovery took weeks and required manual reconciliation.

**SaaS billing incident (2021)**: A migration changed `organization_name` from VARCHAR(500) to VARCHAR(100). Long company names were truncated without warning. Invoices were generated with incomplete names. Customers received bills with garbled business names. Compliance issues ensued.

**Healthcare system EHR**: A hospital reduced `patient_notes` from TEXT to VARCHAR(255) to "improve performance." Thousands of clinical notes were cut off. Doctors had incomplete patient histories. A patient was misdiagnosed due to missing notes. The organization faced HIPAA violations.

## How Failures Happen

```sql
-- Original schema (works fine for years)
CREATE TABLE Customers (
    Id INT PRIMARY KEY,
    Name VARCHAR(255),
    Email VARCHAR(255),
    PhoneNumber VARCHAR(30),
    Address VARCHAR(255)
);

-- Data inserted (fine at the time)
INSERT INTO Customers VALUES
    (1, 'International Ltd', 'contact@intl.example.com', '+1-555-0123 ext. 9999', '1600 Pennsylvania Avenue NW, Washington DC 20500');

-- v2.0 Migration: "Optimize schema, nobody needs long columns"
ALTER TABLE Customers
    ALTER COLUMN Name VARCHAR(50),      -- Was 255, now truncates to 50
    ALTER COLUMN Address VARCHAR(100);   -- Was 255, now truncates to 100

-- Silent data loss happens here! No error.
-- SELECT Name, Address FROM Customers WHERE Id = 1:
-- Name: 'International L' (was 'International Ltd')
-- Address: '1600 Pennsylvania Avenue NW, Washington' (cut off)

-- Application still runs, but data is corrupted
SELECT * FROM Customers WHERE Id = 1;
-- Returns incomplete data without any indication of truncation
```

## Database Behavior

Different databases handle column reduction differently:

| Database | Behavior |
|----------|----------|
| SQL Server | Truncates silently (no error) |
| PostgreSQL | Truncates silently |
| MySQL | Truncates silently (InnoDB, MyISAM) |
| SQLite | Allows, truncates silently |
| Oracle | Allows, truncates silently |

**None of them error by default.** The responsibility is entirely on the developer to:
1. Check for existing data that will be truncated
2. Back up data before running the migration
3. Handle the truncation explicitly or abort the migration

## GauntletCI Detection

GCI0050 detects schema migrations that reduce column size:
1. `ALTER TABLE ... ALTER COLUMN ... VARCHAR(N)` where N < previous length
2. `MODIFY COLUMN` statements (MySQL, Oracle)
3. EF Core migrations: `property.HasMaxLength(newLength)`
4. Dapper migrations, FluentMigrator, etc.

The rule flags these without checking if data exists first. A migration that truncates is high-risk, even if the current data set is empty.

## False Positives

**Not flagged** (safe operations):
```sql
-- Increasing column size - SAFE
ALTER TABLE Customers
    ALTER COLUMN Name VARCHAR(500); -- More space, no truncation risk

-- Changing type to compatible wider type - SAFE
ALTER TABLE Customers
    ALTER COLUMN PhoneNumber VARCHAR(50); -- Wider, no risk

-- Migration with explicit data validation - ACCEPTABLE (but flagged by rule)
-- GCI0050 is intentionally conservative: flags all reductions
-- Developers can suppress with a comment if they've validated
```

**Flagged** (dangerous operations):
```sql
-- Reducing column size without validation - FLAGGED
ALTER TABLE Customers
    ALTER COLUMN Address VARCHAR(100); -- Reduces from 255, risky

-- Truncating TEXT column - FLAGGED
ALTER TABLE Notes
    ALTER COLUMN Content VARCHAR(1000); -- Was TEXT, now limited, risky

-- Multiple simultaneous reductions - FLAGGED
ALTER TABLE Customers
    ALTER COLUMN Name VARCHAR(50),
    ALTER COLUMN Email VARCHAR(50);
```

## Entity Framework Migration Example

```csharp
// DANGEROUS migration (generated by EF)
public partial class TruncateCustomerName : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // This will truncate existing data!
        migrationBuilder.AlterColumn<string>(
            name: "Name",
            table: "Customers",
            type: "varchar(50)",
            maxLength: 50,  // Reduced from 255!
            nullable: false);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AlterColumn<string>(
            name: "Name",
            table: "Customers",
            type: "varchar(255)",
            maxLength: 255,
            nullable: false);
    }
}
```

## When It Fires

GCI0050 fires when:
1. A column size is explicitly reduced in a migration
2. The operation is ALTER COLUMN, MODIFY COLUMN, or HasMaxLength decrease
3. The change is detectable in the diff (before -> after)

It does NOT fire on:
- Adding new columns
- Increasing column size
- Changing column types to compatible larger types
- Deleting columns (different risk, different rule)

## Remediation

**Before** (Risky):
```sql
-- Migration that blindly truncates
ALTER TABLE Customers ALTER COLUMN Address VARCHAR(100);
```

**After** (Safe):
```sql
-- Option 1: Back up data before truncating
BEGIN TRANSACTION;

-- Create backup table
SELECT * INTO Customers_Backup FROM Customers WHERE LEN(Address) > 100;

-- Check if any data would be truncated
IF EXISTS (SELECT 1 FROM Customers WHERE LEN(Address) > 100)
BEGIN
    PRINT 'WARNING: Data truncation detected!';
    PRINT 'Review Customers_Backup table before proceeding.';
    ROLLBACK TRANSACTION;
    RETURN;
END;

-- Safe to proceed
ALTER TABLE Customers ALTER COLUMN Address VARCHAR(100);
COMMIT TRANSACTION;

-- Option 2: Migrate data explicitly
-- Step 1: Add new column
ALTER TABLE Customers ADD Address_New VARCHAR(100);

-- Step 2: Migrate data (truncate and store new values)
UPDATE Customers SET Address_New = LEFT(Address, 100);

-- Step 3: Drop old column
ALTER TABLE Customers DROP COLUMN Address;

-- Step 4: Rename new column
EXEC sp_rename 'Customers.Address_New', 'Address', 'COLUMN';

-- Option 3: Keep the data, extend elsewhere
-- If Address legitimately needs to be longer:
-- Don't reduce the column. Keep it wide.
-- Or split into separate columns (Address_Line1, Address_Line2)
```

## Entity Framework Safe Pattern

```csharp
public partial class SafeMigrationWithValidation : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // Before making the change, we'd run a validation query
        // SELECT COUNT(*) FROM Customers WHERE LEN(Name) > 50
        // If this returns > 0, ABORT the migration

        // For this example, assume validation happened and passed
        
        migrationBuilder.Sql(@"
            IF EXISTS (SELECT 1 FROM Customers WHERE LEN(Name) > 50)
            BEGIN
                RAISERROR('Cannot reduce Name column: data would be truncated', 16, 1);
            END");

        migrationBuilder.AlterColumn<string>(
            name: "Name",
            table: "Customers",
            type: "varchar(50)",
            maxLength: 50,
            nullable: false);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AlterColumn<string>(
            name: "Name",
            table: "Customers",
            type: "varchar(255)",
            maxLength: 255,
            nullable: false);
    }
}
```

## Prevention Strategy

1. **Review migrations before applying**: Check if any column reductions exist
2. **Validate data first**: Run a query to check for values that will be truncated
3. **Back up before migrating**: Always have a recent backup
4. **Archive truncated data**: Keep records of what was removed for compliance
5. **Test in staging**: Never run untested migrations on production
6. **Use explicit validation**: Add migration code that aborts if truncation would occur

## Key Takeaway

**Never reduce column sizes without explicit validation and backup.** Data loss is silent and permanent. Always:
- Check existing data before reducing column size
- Maintain backups before applying the migration
- Consider whether the reduction is necessary (storage savings are rarely worth data loss)
- Document any truncated data for compliance/audit purposes

## Migration Checklist

- [ ] Identified all ALTER COLUMN operations that reduce size
- [ ] Queried production data to verify no truncation will occur
- [ ] Created backup of affected table
- [ ] Added migration code that validates and aborts if needed
- [ ] Tested migration in staging environment
- [ ] Documented any data that may be truncated
- [ ] Obtained approval from product team before applying to production
